# How It Works

## Architecture Overview

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Your Agent     │────▶│  wrapWithSentinel │────▶│  x402 Fetch     │
│  (fetch call)   │     │  (SDK wrapper)   │     │  (payment flow) │
└─────────────────┘     └────────┬─────────┘     └────────┬────────┘
                                 │                        │
                                 │  Before payment        │  After payment
                                 ▼                        ▼
                    ┌────────────────────┐    ┌────────────────────┐
                    │  BudgetManager     │    │  AuditLogger       │
                    │  (evaluate policy) │    │  (log to storage)  │
                    └────────────────────┘    └────────┬──────────┘
                                 │                      │
                                 │  Block if violated   │
                                 ▼                      ▼
                    ┌────────────────────┐    ┌────────────────────┐
                    │  SentinelBudgetError│    │  StorageBackend    │
                    │  (throws)           │    │  (Memory/File/API)  │
                    └────────────────────┘    └────────┬────────────┘
                                                       │
                                                       ▼
                                            ┌────────────────────┐
                                            │  SentinelDashboard │
                                            │  (queries, export) │
                                            └────────────────────┘
```

## Data Flow

1. **Request** — Your agent calls `secureFetch(url)`. The wrapper intercepts the call.

2. **Payment Required** — If the server returns `402 Payment Required`, the x402 client negotiates payment. Sentinel receives the payment context (amount, endpoint, agent, etc.).

3. **Budget Evaluation** — Before payment settles, `BudgetManager.evaluate(context)` runs. If the policy would be violated (e.g., hourly limit exceeded), Sentinel throws `SentinelBudgetError` and the request fails.

4. **Payment** — If allowed, the x402 client completes the payment. The server returns the response with a `PAYMENT-RESPONSE` header.

5. **Audit** — Sentinel extracts settlement data (tx hash, payer, etc.) and calls `AuditLogger.log()`. The record is written to the configured `StorageBackend`.

6. **Hooks** — Optional `afterPayment`, `onBudgetExceeded`, `onAnomaly` hooks run for side effects (logging, alerts, etc.).

## Components

| Component | Role |
|-----------|------|
| **wrapWithSentinel** | Wraps fetch, orchestrates budget + audit |
| **BudgetManager** | Evaluates payments against policy, tracks spend state |
| **AuditLogger** | Writes records to storage, supports query/export |
| **StorageBackend** | Pluggable: Memory, File, or API |
| **SentinelDashboard** | Queries storage for spend reports, agents, alerts |

## SDK vs API

The SDK runs **in-process** with your agent. All budget evaluation and audit logging happens locally. Storage backends can be:

- **Memory** — In-process, no persistence (default)
- **File** — JSONL file on disk
- **API** — Batched POST to a remote API (e.g., api.valeo.money)

The remote API is optional. You can run fully offline with file storage.
